Tema 1

Tema 1 - Optimitzacio de BDR

Has de distingir molt clarament quina eina toca en cada situacio. Si aixo ho controles, la teoria d'aquest tema cau molt millor.

Idea central

L'extensio procedimental afegeix a SQL capacitats de programacio estructurada per reutilitzar consultes, encapsular logica i reaccionar automaticament als canvis.

variables condicionals bucles funcions procediments triggers

Mapa de decisio

flowchart TD
    Q["Necessites fer..."] --> A{"Nomes consultar?"}
    A -->|Si| B{"Cal passar
parametres?"}
    B -->|No| V["VIEW"]
    B -->|Si| F["FUNCTION"]
    A -->|No| C{"Ha de reaccionar
automaticament?"}
    C -->|No| P["PROCEDURE"]
    C -->|Si| T["TRIGGER"]
    T --> T1["INSTEAD OF"]
    T --> T2["AFTER"]

Avantatges

  • reutilitzacio de codi
  • millor manteniment
  • mes seguretat
  • millor rendiment si la logica s'executa al SGBD
  • automatitzacio de validacions i camps calculats

Inconvenients o limits

  • la sintaxi varia segons el SGBD
  • les vistes tenen restriccions
  • massa triggers compliquen el manteniment
  • si tota la logica queda al servidor, costa mes depurar

Sintaxi clau

CREATE VIEW v_alumnes AS
SELECT nom, cognoms
FROM alumnes;
CREATE FUNCTION f_alumnes_grau(@grau VARCHAR(100))
RETURNS TABLE
AS
RETURN (
  SELECT * FROM alumnes
  WHERE grau = @grau
);
CREATE PROCEDURE sp_exemple(@dni CHAR(9))
AS
BEGIN
  DELETE FROM alumnes
  WHERE dni = @dni;
END;
CREATE TRIGGER trg_exemple
ON alumnes
AFTER INSERT
AS
BEGIN
  PRINT('Nou alumne');
END;
Frase per memoritzar view consulta, function retorna, procedure actua, trigger reacciona

Variables, IF, WHILE i errors

DECLARE @x INT;
SET @x = 10;

IF @x > 5
BEGIN
  PRINT('gran');
END
ELSE
BEGIN
  PRINT('petit');
END;

WHILE @x > 0
BEGIN
  SET @x = @x - 1;
END;

Limits importants de vistes i funcions

  • una VIEW no accepta parametres
  • si hi ha COUNT, AVG o camps calculats, cal alias
  • ORDER BY dins d'una vista te restriccions
  • les FUNCTION del temari funcionen com una vista amb parametres

Procediments i RAISERROR

RAISERROR('text_error', 16, 1);
RETURN;

EXEC sp_exemple '12345678A';

`RAISERROR` mostra un error; `RETURN` talla l'execucio si cal.

Triggers: que no pots oblidar

  • INSTEAD OF: substitueix la instruccio
  • AFTER: s'executa despres
  • treballen amb les taules virtuals inserted i deleted
  • son molt utils per validar dades i recalcular redundancies
Tema 2

Tema 2 - SQL cap a MongoDB

A classe heu treballat sobretot la migracio del model relacional cap a un model documental. La clau es pensar sempre en l'equivalencia SQL -> collection/document.

Idea central

MongoDB es documental. Guarda la informacio en documents JSON/BSON dins de col.leccions. Va be quan necessites flexibilitat, dades heterogenies i escalabilitat horitzontal.

esquema flexible consultes rapides escalabilitat horitzontal

Equivalencies SQL vs MongoDB

SQLMongoDB
taulacollection
filadocument
columnafield
JOIN$lookup
SELECTfind o aggregate

ACT1 ex. 1 - Passar d'una taula SQL a MongoDB

flowchart TD
    A["BD relacional"] --> B["use('Hotel')"]
    B --> C["cada taula -> collection"]
    C --> D["cada fila -> document"]
    D --> E["cada columna -> field"]
    E --> F["insertOne / insertMany"]

Exemple basic

use("Hotel");

db.ciutat.insertOne({
  nom: "Barcelona",
  nHabitants: 2000000,
  port: "Si"
});

Avantatges

  • esquema flexible
  • facil afegir camps nous
  • molt bo amb dades semiestructurades
  • mes facil de distribuir

Inconvenients

  • mes redundancia
  • consistencia menys estricta
  • no hi ha claus foranies enforced com a SQL
  • consultes complexes es pensen diferent

Com decidir la comanda

flowchart TD
    Q["Vols consultar"] --> A{"Nomes filtres i projeccio?"}
    A -->|Si| F["find / findOne"]
    A -->|No| B{"Hi ha group by, calculs o joins?"}
    B -->|Si| AG["aggregate"]
    B -->|No| F
    AG --> L["$lookup"]
    L --> U["$unwind si cal desplegar arrays"]

CRUD

insertOne insertMany updateOne updateMany deleteOne deleteMany

Operadors clau

$gt $lt $gte $lte $in $nin $or $and $regex

Pipeline tipica

$match $sort $group $project $limit

DDL i DML a MongoDB

  • als apunts no hi ha un DDL fix com a SQL
  • les col.leccions i els camps apareixen quan hi insereixes dades
  • la part operativa forta es el DML: inserir, modificar, eliminar i consultar

SQL -> Mongo per condicions frequents

BETWEEN
{ camp: { $gte: a, $lte: b } }

IN
{ camp: { $in: [v1, v2] } }
LIKE
{ nom: { $regex: ".*ona" } }

ORDER BY / LIMIT
.sort({ camp: 1 }).limit(10)

Agregacions que solen caure

$count $sum $avg $max $min distinct

Si has de comptar, sumar, fer mitjana o agrupar, gairebe segur que toca `aggregate`.

JOIN i UNION equivalents

  • $lookup: equivalent conceptual de `JOIN`
  • $unwind: desplega arrays retornats pel lookup
  • $unionWith: equivalent conceptual de `UNION`
{
  $lookup: {
    from: "empresa",
    localField: "nom",
    foreignField: "seuSocial",
    as: "dadesEmpresa"
  }
}
Tema 3

Tema 3 - Bases de dades distribuides

Aquest tema sembla llarg, pero realment gira sobretot al voltant de dues idees: fragmentacio i transparencia.

Idea central

Una BDD distribuida es una sola base de dades repartida entre diversos nodes. Idealment, l'usuari no hauria de notar que esta distribuida.

Tipus

  • Top-Down: es dissenya des de zero i despres es distribueix.
  • Bottom-Up: s'integra el que ja existeix.

Fragmentacio

flowchart TD
    F["Fragmentacio"] --> H["Horitzontal"]
    H --> H1["Primaria
condicio sobre la mateixa taula"]
    H --> H2["Derivada
condicio sobre taula relacionada"]
    F --> V["Vertical
divideix atributs"]
    V --> PK["clau primaria a tots els fragments verticals"]
    F --> M["Mixta"]
    M --> M1["HV
horitzontal i despres vertical"]
    M --> M2["VH
vertical i despres horitzontal"]
    M --> M3["En celles
horitzontal i vertical alhora"]
    F --> R["Regles"]
    R --> R1["Completesa"]
    R --> R2["Disjuntivitat"]
    R --> R3["Reconstruccio"]

Les 3 paraules que has de dir sempre

completesa disjuntivitat reconstruccio

Si et pregunten un esquema de fragmentacio i no justifiques aquestes tres, la resposta queda coixa.

Horitzontal

  • Primaria: la condicio es sobre un atribut de la mateixa taula.
  • Derivada: la condicio es sobre una taula relacionada.
  • La reconstruccio es fa amb UNION.

Vertical

  • Divideix la taula per atributs.
  • La clau primaria ha d'apareixer a tots els fragments.
  • La reconstruccio es fa amb JOIN.

Mixta

HV VH en celles

Combina fragmentacio horitzontal i vertical. En l'examen sol caure quan hi ha criteris de ciutat, perfil, departament o tipus d'usuari alhora.

Com es reconstrueix la taula original

Si la fragmentacio es horitzontal reconstruccio amb UNION entre fragments
Si la fragmentacio es vertical reconstruccio amb JOIN per la clau primaria
Si la fragmentacio es mixta normalment fas primer les reconstruccions locals de les fulles i al final combines UNION i JOIN

Avantatges

  • dades mes a prop d'on s'utilitzen
  • millor disponibilitat
  • mes fiabilitat
  • paral.lelisme
  • facilitat d'expansio

Inconvenients

  • mes complexitat
  • consultes mes lentes per la xarxa
  • problemes de consistencia i replicacio
  • transaccions distribuides mes dificils

Transparencia

flowchart TD
    T["Transparencia"] --> T1["Fragmentacio
veu la taula global"]
    T --> T2["Ubicacio
coneix fragments, no nodes"]
    T --> T3["SGBD local
coneix fragments i nodes"]
    T --> T4["No transparencia
l'usuari ho ha de saber tot"]
    T1 --> A["mes facil per l'usuari"]
    T3 --> B["mes feina per l'usuari"]
    T4 --> C["maxima carrega per l'usuari"]

Replicacio i consistencia

  • La replicacio millora disponibilitat i seguretat.
  • El problema principal es actualitzar copies sense inconsistencies.
  • Consistencia immediata: totes les copies s'actualitzen al moment.
  • Consistencia diferida: les copies s'actualitzen mes tard.

Processament i transaccions

  • GQP: processador global de consultes.
  • LQP: processador local de cada node.
  • El sistema calcula cost de subconsultes i cost de xarxa.
  • 2PC: protocol Two-Phase Commit per coordinar transaccions distribuides.

Part B - SQL cap a XML

Idea central

XML serveix per integrar dades entre sistemes heterogenis. En el vostre ACT2, l'arbre es hotel -> usuari -> client -> reserva -> habitacio.

flowchart TD
    H["hotel"] --> U["usuari"]
    U --> C["client"]
    C --> R["reserva"]
    R --> HB["habitacio"]

Regles basiques de XML

  • ha de tenir node arrel
  • tots els nodes s'han de tancar
  • han d'estar ben niuats
  • XML diferencia majuscules i minuscules
  • els atributs han de tenir valor
  • els valors van entre cometes

Versio, codificacio i caracters especials

  • als apunts surt tipicament XML 1.0 amb UTF-8
  • al vostre ACT2 es demana explicitament XML 2.0 amb UTF-8
  • < -> &lt;
  • > -> &gt;
  • & -> &amp;

FOR XML - RAW, AUTO, PATH

flowchart TD
    SQL["Dades SQL"] --> X["FOR XML"]
    X --> RAW["RAW
cada fila = row"]
    X --> AUTO["AUTO
jerarquia automatica segons JOIN"]
    X --> PATH["PATH
control total"]
    PATH --> P1["@alias -> atribut"]
    PATH --> P2["alias normal -> element"]
    X --> E["ELEMENTS"]
    X --> R["ROOT('nom')"]

RAW

Cada fila del resultat es converteix en un row. Va be per exportacio simple.

SELECT ...
FOR XML RAW;

AUTO

La jerarquia surt de l'ordre del FROM i els JOIN. Tens control limitat.

SELECT ...
FOR XML AUTO;

PATH

La important per examen. Decideixes noms de nodes i atributs.

SELECT codi AS '@codi'
FOR XML PATH('node');

Opcions addicionals que has de saber

ELEMENTS
SELECT ...
FOR XML RAW, ELEMENTS;
ROOT
SELECT ...
FOR XML AUTO, ROOT('hotel');

`ELEMENTS` converteix camps en elements; `ROOT('nom')` afegeix el node arrel.

Patro per construir XML jerarquic

SELECT ...
(
  SELECT ...
  (
    SELECT ...
    FOR XML PATH('reserva'), TYPE
  )
  FOR XML PATH('client'), TYPE
)
FOR XML PATH('usuari'), ROOT('hotel');
Part C - XPath

Mapa XPath

flowchart TD
    X["XPath"] --> X1["//node"]
    X --> X2["@atribut"]
    X --> X3["[condicio]"]
    X --> X4[".."]
    X --> F["Funcions"]
    F --> F1["count"]
    F --> F2["sum"]
    F --> F3["avg"]
    F --> F4["max / min"]
    F --> F5["distinct-values"]
    F --> F6["contains"]

Com pensar una consulta

  1. situa't al node on hi ha la condicio
  2. aplica la condicio
  3. selecciona els atributs que vols
  4. puja o baixa per l'arbre si cal

Expressions basiques

//node //node/fill //node/.. //node/@atribut //node/@*

Equivalencies mentals amb SQL

  • distinct-values fa el paper de `DISTINCT`
  • contains fa el paper de `LIKE`
  • @att = (v1, v2) fa el paper de `IN`
  • count / avg / sum / max / min son les agregacions tipiques

Patrons que s'han de saber

//habitacio/@codi
count(//reserva)
avg(//reserva/@preuTotal)
distinct-values(//habitacio/@codi)
//reserva[@estat="realitzada"]/@numReserva
//node[contains(@camp,"text")]

Altres patrons utiles

//node[@att = ("A", "B")]/@att
//node[@preu > 100]/@codi
//node[@att = max(//node/@att)]/@nom
//node[@x and @y]/@*
